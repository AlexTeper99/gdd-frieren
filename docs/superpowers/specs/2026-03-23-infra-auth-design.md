## Task 1.1 — Infrastructure

### Vercel Project

- `vercel link` in repo root → creates/links project `habit-quest-mocha`
- Framework detection: Next.js 16 (automatic)
- Domain: `habit-quest-mocha.vercel.app`

### Vercel Postgres

- `vercel postgres create` → DB `habit-quest-db`
- Auto-generates env vars: `POSTGRES_URL`, `POSTGRES_URL_NON_POOLING`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DATABASE`, `POSTGRES_HOST`
- Driver: `@neondatabase/serverless` (already in package.json)

### Resend

- Manual signup at resend.com → create API key
- Free tier: 100 emails/day, shared domain (`onboarding@resend.dev`)
- Sufficient for 2 users

### Env Vars

```
# Auto-generated by Vercel Postgres
POSTGRES_URL=...
POSTGRES_URL_NON_POOLING=...

# Manual
AUTH_SECRET=<generated with `npx auth secret`>
AUTH_ALLOWED_EMAILS=email1@...,email2@...
AUTH_RESEND_KEY=<from Resend dashboard>
ANTHROPIC_API_KEY=<for later phases>
```

- `vercel env add` for each manual variable
- `vercel pull` to download all to `.env.local`

### Drizzle Config

- `drizzle.config.ts` pointing to `POSTGRES_URL`
- Driver: neon-http via `@neondatabase/serverless`
- Note: `.env.local.example` currently uses `DATABASE_URL` — update it to `POSTGRES_URL` to match Vercel auto-generated names

### Verification

- `vercel deploy` — deploys empty, DB connected, no errors
- `pnpm db:push` — confirms Drizzle can connect to DB

## Task 1.2 — Auth.js + Magic Links + Proxy

### Auth.js Config (`lib/auth/index.ts`)

- **Provider:** `Resend` from `next-auth/providers/resend` (not the `resend` package directly). Config: `from: "onboarding@resend.dev"`
- **Adapter:** Drizzle adapter (`@auth/drizzle-adapter`) — tables must be explicitly defined in `lib/db/schema.ts` (not auto-generated)
- **Whitelist:** `signIn` callback validates email against `AUTH_ALLOWED_EMAILS` env var. If not in list → reject login

### Auth.js Tables (Drizzle adapter)

| Table | Purpose |
|---|---|
| `users` | id, name, email, emailVerified, image |
| `accounts` | Provider linkage |
| `sessions` | Session tokens |
| `verification_tokens` | Magic link tokens (short TTL, auto-cleanup) |

Note: `users` table will be extended in Phase 1.5 with GDD fields (archetype, character_name, etc). For now, only what Auth.js requires.

### Magic Link Flow

1. User goes to `/login` → enters email
2. Auth.js validates against whitelist → if OK, Resend sends magic link from `onboarding@resend.dev`
3. User clicks link → `/api/auth/callback/resend` (Auth.js built-in callback)
4. Auth.js verifies token → creates session → redirect to `/`

### proxy.ts (project root: `proxy.ts`, NOT `app/proxy.ts`)

Per Next.js 16 docs: proxy.ts must be at project root, same level as `app/`.

- **Matcher:** `config.matcher` excludes `/login`, `/verify`, `/api/auth/*`, static assets
- **Auth check:** Optimistic cookie check only (does session cookie exist?). Does NOT call `auth()` — that's a DB call, too heavy for proxy. Real session validation deferred to `lib/dal.ts` → `verifySession()` inside server components and server actions
- **No session cookie** → redirect to `/login`
- Next.js 16: exports named `proxy` function (not middleware)
- Runtime: Node.js (not Edge)

Note: architecture spec shows `app/proxy.ts` — that's wrong, will be corrected in task 0.3 (arch spec v2).

### Pages

| Route | Purpose |
|---|---|
| `app/(auth)/login/page.tsx` | Form: email input + "Send magic link" button |
| `app/(auth)/verify/page.tsx` | "Check your email" informational page (actual token verification handled by `/api/auth/callback/resend`) |
| `app/(app)/page.tsx` | Placeholder "El Mundo" showing "Welcome, {user.email}" |

### Verification

- Login with whitelisted email → receives magic link → active session → sees El Mundo
- Login with non-whitelisted email → error, no email sent
- Access `/` without session → redirect to `/login`
- Deploy to Vercel works end-to-end
